1. Resumen ejecutivo

1.1 Propósito del documento

Este documento define la arquitectura completa del sistema que da soporte al portal técnico personal/profesional de Rolando Luis Martin.

El sistema no debe entenderse como un simple sitio web estático ni como un blog personal convencional. Su diseño responde a una necesidad más amplia: construir una plataforma de conocimiento técnico modular, estructurada, versionable y mantenible, capaz de integrar:

  • artículos técnicos de largo formato,

  • documentación navegable,

  • fichas de proyectos,

  • contenido institucional,

  • internacionalización,

  • sistema de temas visuales,

  • navegación semántica,

  • automatización editorial,

  • y gobernanza técnica del crecimiento del sistema.

1.2 Naturaleza del sistema

Este proyecto se clasifica como:

  • Sistema editorial técnico

  • Plataforma de publicación estructurada

  • Motor personal de conocimiento técnico

  • Proyecto de ingeniería informática orientado a documentación, publicación y navegación semántica del conocimiento

En términos conceptuales, el sistema se define como:

un mapa de pensamiento técnico publicado mediante una arquitectura modular y reproducible.

1.3 Objetivo estratégico

El objetivo estratégico del sistema es crear una plataforma que permita representar públicamente no solo el conocimiento técnico del autor, sino también su forma de pensar, estructurar, relacionar y documentar sistemas complejos.

La plataforma debe permitir:

  • publicar conocimiento con calidad editorial,

  • mostrar proyectos como sistemas, no como fichas aisladas,

  • conectar conceptos, procedimientos y arquitecturas,

  • crecer sin entropía,

  • mantenerse en GitHub Pages con simplicidad operativa,

  • y servir como activo profesional de largo plazo.

3. Principios de ingeniería del sistema

3.1 Principio de modularidad

Todo el sistema se construye a partir de módulos con responsabilidades delimitadas.

Las piezas del sistema deben ser:

  • comprensibles,

  • reemplazables,

  • escalables,

  • y desacopladas en lo posible.

La modularidad aplica a:

  • layouts,

  • componentes,

  • contenido,

  • datos,

  • estilos,

  • scripts,

  • automatización.

3.2 Principio de separación de responsabilidades

El sistema separa estrictamente:

  • contenido: conocimiento y cuerpo editorial,

  • metadatos: descripción estructural del contenido,

  • layout: organización espacial,

  • componentes: piezas reutilizables de interfaz,

  • tema: apariencia visual,

  • idioma: textualización y localización,

  • render: ensamblaje final de página,

  • automatización: generación, validación y build.

Ninguna capa debe asumir las responsabilidades de otra salvo por composición explícita.

3.3 Principio de identidad lógica estable

Cada entidad de contenido debe tener un identificador lógico único y estable, independiente de:

  • idioma,

  • slug,

  • estructura visual,

  • cambios editoriales,

  • evolución del diseño.

Ejemplo:

id = windows-update-offline

Ese identificador permanece aunque el artículo:

  • cambie de título,

  • cambie de slug,

  • se traduzca,

  • se relacione con otros proyectos,

  • o se mueva de sección.

3.4 Principio de build reproducible

El sistema debe poder reconstruirse íntegramente desde el repositorio.

Esto implica:

  • configuración declarativa,

  • contenido fuente claro,

  • plantillas estables,

  • scripts de generación controlados,

  • ausencia de dependencias ocultas,

  • y capacidad de rehacer la salida final de manera determinista.

3.5 Principio de gobernanza antes que improvisación

El crecimiento del sistema no se basa en excepciones locales, sino en reglas.

Toda nueva decisión debe responder a:

  • coherencia arquitectónica,

  • justificación funcional,

  • impacto sistémico,

  • mantenibilidad futura.

3.6 Principio de automatización del trabajo repetitivo

El sistema automatiza:

  • scaffolding,

  • validación,

  • cálculo de derivados,

  • generación de índices,

  • build,

  • publicación.

No automatiza el criterio técnico ni la autoría intelectual.

4. Visión conceptual del sistema

4.1 Definición general

El sistema puede describirse mediante la siguiente fórmula conceptual:

Página = Ruta + Contexto + Entidad + Shell + Layout + Componentes + Tema + Estado

Donde:

Ruta

Define:

  • idioma,

  • sección,

  • slug,

  • punto de entrada.

Contexto

Define:

  • idioma activo,

  • diccionario UI,

  • tema activo,

  • preferencias del usuario,

  • configuración global.

Entidad

Representa la pieza de contenido:

  • article,

  • project,

  • doc,

  • page.

Shell

Estructura global persistente:

  • header,

  • navegación principal,

  • selector de idioma,

  • selector de tema,

  • footer.

Layout

Define la organización espacial de la página.

Componentes

Renderizan partes concretas de interfaz dentro del layout.

Tema

Aplica la apariencia visual sin alterar estructura ni contenido.

Estado

Gestiona comportamiento interactivo y preferencias:

  • tema actual,

  • colapsado de TOC,

  • menú móvil,

  • etc.

4.2 Definición operativa del sistema

El sistema completo puede describirse como:

una plataforma estática de conocimiento técnico basada en entidades estructuradas, ensambladas a través de layouts y componentes, tematizadas mediante tokens visuales, internacionalizadas mediante variantes idiomáticas, y publicadas por un pipeline reproducible en GitHub Pages.

6. Arquitectura lógica

6.1 Shell global

Definición

El shell es la estructura persistente y transversal del portal.

Responsabilidades

  • presentar la cabecera global,

  • mostrar navegación principal,

  • exponer selector de idioma,

  • exponer selector de tema,

  • incluir footer,

  • alojar el layout actual.

Componentes típicos del shell

  • c-site-header

  • c-main-nav

  • c-language-switcher

  • c-theme-switcher

  • c-site-footer

Restricciones

El shell no contiene contenido editorial específico.
No debe decidir layouts ni renderizar entidades por sí mismo.

6.2 Layouts

Definición

Un layout es una plantilla estructural que organiza regiones de la página.

Layouts iniciales oficiales

  • l-simple

  • l-article

  • l-docs

  • l-showcase

Regla fundamental

El layout organiza espacio, no contenido.

Ejemplo: l-article

Regiones:

  • top

  • reader

  • body

    • main

    • aside

  • bottom

Uso:

  • artículos largos,

  • tutoriales,

  • análisis técnicos.

6.3 Componentes

Definición

Unidad reutilizable de interfaz con responsabilidad concreta.

Taxonomía

Globales

  • header

  • nav

  • footer

  • switchers

Editoriales

  • article-header

  • article-meta

  • prose

  • toc

  • reader-bar

  • callout

  • code-block

Navegación

  • breadcrumb

  • sidebar-nav

  • prev-next

Descubrimiento

  • article-card

  • project-card

  • related-posts

Interactivos

  • copy button

  • collapsibles

  • tabs

  • accordions

Regla de diseño

Un componente:

  • hace una cosa,

  • expone contrato claro,

  • usa tokens,

  • consume diccionario si tiene UI textual,

  • define su comportamiento responsive.

  • spacing,

  • radios,

  • sombras,

  • familias tipográficas.

Tokens semánticos

Valores consumidos por la UI:

  • --bg-page

  • --bg-panel

  • --text-main

  • --text-muted

  • --border-default

  • --accent

  • --link

  • --code-bg

8.2 Temas oficiales iniciales

  • classic

  • graphite

  • solar

8.3 Regla fundamental

8.4 Persistencia del tema

9. Modelo de contenido

9.1 Entidades oficiales

  • article

  • project

  • doc

  • page

9.2 Contrato base común

Toda entidad debe tener al menos:

  • id

  • type

  • status

  • layout

  • createdAt

  • updatedAt

  • translations

  • relations

9.3 Estados editoriales globales

  • draft

  • published

  • archived

9.4 Estados por idioma

  • missing

  • draft

  • published

9.5 Variantes por idioma

Cada entidad contiene un mapa translations.

Ejemplo conceptual:


                    "translations": {

                      "es": {

                        "status": "published",

                        "slug": "actualizar-windows-11-offline",

                        "title": "Actualizar Windows 11 offline",

                        "description": "Guía técnica para actualizar equipos sin dependencia de red."

                      },

                      "en": {

                        "status": "missing"

                      }

                    }

9.6 Cuerpo del contenido

El cuerpo editorial se almacena desacoplado de los metadatos, típicamente en Markdown:

  • body.es.md

  • body.en.md

13. Sistema editorial

13.1 Ciclo editorial


                    Idea

                    → Alta de entidad

                    → Redacción

                    → Revisión

                    → Publicación

                    → Traducción

                    → Actualización

                    → Archivado

13.2 Regla base

Toda pieza nueva nace desde una plantilla, no desde HTML libre.

13.3 Condiciones de publicabilidad

Una entidad es publicable si:

  • tiene identidad estable,

  • metadatos mínimos completos,

  • layout asignado,

  • cuerpo editorial suficiente,

  • navegación resoluble,

  • integración en hubs o relaciones.

un motor personal de conocimiento técnico, versionado, modular, reproducible y mantenible.
