Conversor de Información Técnica en Activos Reutilizables

1. Introducción

En los entornos de Soporte Técnico, Programación y Sistemas, el conocimiento no suele faltar.
Lo que falta es estructura, reutilización y continuidad.

Soluciones técnicas valiosas aparecen cada día en:

  • chats con IAs,
  • foros,
  • correos,
  • incidencias resueltas,
  • búsquedas puntuales.

Sin embargo, este conocimiento suele quedar:

  • disperso,
  • mal formateado,
  • difícil de reutilizar,
  • dependiente de la memoria individual.

Este proyecto nace para resolver ese problema de raíz.

2. Slogan / Principio Rector

Convertir conocimiento técnico disperso en activos reutilizables.

Este principio guía todas las decisiones de diseño, arquitectura y alcance del sistema.

3. Objetivo del Sistema

Objetivo principal

Dado un HTML caótico (copiado de chats, webs, IAs, foros),
extraer y normalizar únicamente el contenido semántico útil
eliminando atributos, scripts, estilos, basura y ruido,
y adaptarlo automáticamente:

  • a un layout corporativo unificado,
  • a un sistema de multi‑skins definido centralmente,
  • a tipologías de artículos técnicas,
  • con metadatos estructurados y exportables.

Objetivo estratégico

Transformar soluciones aisladas en activos documentales, mantenibles y reutilizables por toda la organización.

4. Alcance del Conversor

El sistema no pretende:

  • redactar contenido nuevo,
  • inventar soluciones,
  • sustituir el criterio técnico.

El sistema sí pretende:

  • limpiar,
  • estructurar,
  • normalizar,
  • presentar,
  • reutilizar.

5. Arquitectura Conceptual (visión por capas)

El conversor se concibe como un pipeline de transformación, compuesto por capas bien definidas.

5.1 Capa de Ingesta (Input)

Responsabilidad: recibir información sin interpretarla.

Fuentes admitidas:

  • HTML de chats con IA
  • HTML copiado desde navegador
  • HTML de foros técnicos
  • Ficheros .html guardados localmente

Características del input:

  • desordenado
  • inconsistente
  • con dependencias externas
  • con ruido visual y técnico

5.2 Capa de Limpieza Semántica (Sanitization)

Esta es la pieza crítica del sistema.

Responsabilidades:

  • Eliminar completamente:
    • <script>
    • <style>
    • <iframe>
    • comentarios
  • Eliminar todos los atributos:
    • class, id, style, data-*, etc.
  • Conservar:
    • etiquetas estructurales (h1–h6, p, ul, li, pre, code, etc.)
    • el contenido textual íntegro
  • Extraer únicamente el <body>

Resultado:
HTML limpio, neutro, portable y semánticamente coherente.

5.3 Capa de Normalización Editorial

Aquí el contenido se interpreta, no se reescribe.

Objetivo: adaptar el HTML limpio a un modelo editorial común.

Ejemplos de normalización:

  • un único <h1> como título del artículo
  • <h2> como secciones
  • <pre><code> como bloques técnicos
  • listas como procedimientos o checklists

Esto permite:

  • coherencia entre artículos
  • facilidad de búsqueda
  • reutilización futura
  • exportación homogénea

5.4 Capa de Plantilla (Layout Corporativo)

El contenido normalizado se inyecta en una plantilla estable.

Características del layout:

  • estructura fija
  • semántica clara
  • compatible con web, KB y PDF

El layout no depende del contenido.
El contenido no rompe el layout.

5.5 Capa de Skins / Multi‑Tema

Sistema de identidad visual desacoplado del contenido.

Principios:

  • temas definidos centralmente (GitHub)
  • uso de variables CSS / design tokens
  • mismo contenido → múltiples apariencias

Casos de uso:

  • Soporte IT (modo oscuro)
  • Cliente final (modo claro)
  • Auditoría / documentación formal

5.6 Capa de Metadatos

Cada artículo generado es un activo documental, no solo HTML.

Metadatos previstos:

  • título
  • categoría (Soporte, Windows, Programación, Herramientas…)
  • subcategoría
  • nivel (Junior / Senior)
  • tags
  • origen del conocimiento
  • fecha / versión

Estos metadatos permiten:

  • indexación
  • trazabilidad
  • reutilización sistemática

5.7 Capa de Exportación

Un único origen, múltiples destinos:

  • HTML final para web
  • PDF con identidad visual
  • Base de Conocimiento corporativa
  • Repositorios documentales

6. Principios de Diseño

El sistema se rige por los siguientes principios:

  • Separación de responsabilidades
  • Contenido ≠ presentación
  • Reutilización por defecto
  • Escalabilidad sin deuda técnica
  • Automatizar lo repetitivo, preservar el criterio humano

7. Valor para Soporte Técnico

Este conversor permite:

  • reducir tiempo de documentación
  • evitar duplicar soluciones
  • elevar la calidad documental
  • crear memoria técnica colectiva
  • transformar incidencias en conocimiento reutilizable

8. Visión a Medio Plazo

Este sistema sienta las bases para:

  • una KB viva
  • estándares documentales
  • herramientas internas orientadas a cliente
  • madurez técnica del área de soporte

9. Conclusión

Este proyecto no trata de HTML.

Trata de:

  • ordenar conocimiento
  • preservar soluciones
  • hacer crecer a las personas a través de sistemas

Convertir conocimiento técnico disperso en activos reutilizables
no es un eslogan: es una estrategia.
