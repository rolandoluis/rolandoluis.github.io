Nota

Útil para entornos de Laboratorio de Soporte IT, donde la dinámica, la precisión y la productividad, van de la mano.

Objetivo

Unir, en condiciones de laboratorio, una PC al Dominio de Active Directory, utilizando Comandos de Terminal

Contexto

En la administración avanzada de Windows, unir un equipo a un dominio mediante la línea de comandos permite agilizar despliegues y facilitar el soporte remoto. Este documento detalla los procedimientos recomendados utilizando PowerShell y el símbolo del sistema (CMD).

📋 Requisitos Previos

Antes de iniciar, asegúrese de cumplir con los siguientes puntos para evitar errores de autenticación o resolución:

  • Conectividad: El equipo debe tener línea de vista (ping) con el Controlador de Dominio (DC).

  • DNS: Las DNS del adaptador de red deben apuntar a las direcciones IP de los controladores de dominio.

  • Privilegios: Se requieren credenciales de un usuario con permisos para unir equipos al dominio (Domain Admin o Delegated User).

  • Ejecución: Ambas consolas deben ejecutarse con privilegios de Administrador.

Desarrollo

🛠️ Método 1: PowerShell (Recomendado)

PowerShell ofrece la forma más moderna y eficiente de gestionar esta tarea, permitiendo incluso la ubicación específica del equipo en el Active Directory.

Unión Estándar

Para unir el equipo al dominio y reiniciar automáticamente:

                    
                        Add-Computer -DomainName "dominio.com" -Restart

Unión a una Unidad Organizativa (OU) Específica

Si el flujo de trabajo requiere que el objeto de equipo se cree en una ruta específica dentro del AD:

                    
                        Add-Computer -DomainName "dominio.com" -OUPath "OU=Equipos,DC=dominio,DC=com" -Restart

💻 Método 2: Símbolo del Sistema (CMD)

Este método es útil en entornos donde PowerShell esté restringido o se utilicen scripts de legado (.bat / .cmd). Se utiliza la herramienta netdom.

Comando de Ejecución:

                    netdom join %COMPUTERNAME% /domain:dominio.com /userd:UsuarioAdmin /passwordd:*
Tip

[!IMPORTANT] Nota sobre seguridad: El uso del asterisco (*) en el parámetro /passwordd es una buena práctica de seguridad, ya que el sistema solicitará la contraseña en una línea oculta en lugar de escribirla en texto plano en el historial de la consola.

✅ Consideraciones Finales

  1. Reinicio: Ambos métodos (usando el parámetro -Restart en PowerShell o por naturaleza del cambio en CMD) requieren un reinicio para que el equipo cargue las políticas de grupo (GPOs) y reconozca la nueva relación de confianza.

  2. Confirmación: Una vez reiniciado, puede verificar el estado del dominio ejecutando el comando whoami /fqdn o revisando las Propiedades del Sistema.

Advertencia

¡¡¡Ufff, me ha salido un mensaje de "Trust Relationship"!!!.

Ese error es el "pan de cada día" en soporte técnico, pero cuando aparece por duplicidad de nombre, puede ser un pequeño dolor de cabeza si no se limpia bien el rastro.

🔍 Resolución de Errores de Relación de Confianza (Trust Relationship)

El error "La relación de confianza entre esta estación de trabajo y el dominio principal falló" ocurre cuando el secreto compartido (una contraseña de cuenta de equipo que cambia automáticamente cada 30 días) se desincroniza entre el Active Directory (AD) y el equipo local.

⚠️ Causas Comunes

  • Nombres Duplicados: Como te sucedió hoy, si unes un equipo con un nombre que ya existe en el AD, el servidor sobrescribe el objeto anterior, "rompiendo" la conexión del equipo original.

  • Restauración de Snapshots: En máquinas virtuales, volver a un estado anterior puede revertir la contraseña del equipo a una versión que el AD ya no reconoce.

  • Inactividad Prolongada: Si un equipo está apagado meses, el AD puede marcar la cuenta como expirada o haber cambiado la clave sin que el equipo se enterara.

🛠️ Cómo Solucionar la Duplicidad de Nombre

Si el error fue por un nombre idéntico, el alcance es que ninguno de los dos equipos podrá iniciar sesión con cuentas de dominio hasta que se resuelva.

Paso 1: Limpieza en el Controlador de Dominio (AD)

Antes de intentar unir el equipo de nuevo, es vital limpiar el rastro:

  1. Busca el nombre del equipo en Usuarios y Equipos de Active Directory.

  2. Haz clic derecho y selecciona "Restablecer cuenta" (esto pone la contraseña a un estado inicial) o, preferiblemente, Eliminar el objeto si vas a hacer una unión limpia.

Paso 2: Reparación desde el Equipo (PowerShell)

No siempre es necesario sacar el equipo del dominio y volverlo a meter (el método largo). Puedes intentar reparar el canal seguro con este comando:

                    Test-ComputerSecureChannel -Repair -Credential (Get-Credential)

Este comando te pedirá credenciales de administrador de dominio y forzará la sincronización de la contraseña sin necesidad de reiniciar.

Paso 3: El "Truco del Grupo de Trabajo" (Si lo anterior falla)

Si la reparación falla por la duplicidad:

  1. Cambia el equipo a un Grupo de Trabajo (ej. WORKGROUP).

  2. Reinicia.

  3. Cambia el nombre del equipo a uno único que no exista en la red.

  4. Vuelve a unirlo al dominio usando los comandos de la sección anterior.

💡 Alcance del Problema

Cuando la confianza se rompe:

  • Usuarios de dominio: No pueden iniciar sesión (verán el error de confianza).

  • Usuarios locales: Siguen funcionando (es tu vía de entrada para arreglarlo).

  • GPOs: Dejan de aplicarse, dejando al equipo vulnerable o sin configuraciones corporativas.

Tip

Pero... ¿Son muchos pasos manuales y la puedes liar?. El siguiente script te puede ayudar a automatizar el proceso

Este script realiza tres acciones críticas en una sola ejecución: cambia el nombre del equipo, lo une al dominio en la ruta (OU) que elijas y solicita el reinicio necesario.

📜 Script: Cambio de Nombre y Unión al Dominio

Copia este código en un archivo .ps1 o ejecútalo directamente en una consola de PowerShell como Administrador.

                    # 1. Definición de variables (Edita estos campos)
$nuevoNombre = "PC-VENTAS-01"
$dominio = "dominio.com"
$ouPath = "OU=Equipos,DC=dominio,DC=com" # Opcional: Déjalo en blanco si no usas OUs específicas
# 2. Credenciales (Abrirá una ventana emergente segura)
$credenciales = Get-Credential <>-UserName "admin_dominio" -Message "Introduce credenciales de Administrador de Dominio"
Write-Host "Iniciando proceso de cambio de nombre y unión..." -ForegroundColor Cyan
try {
    # 3. Ejecución del comando principal
    Add-Computer -DomainName $dominio `
                 -NewName $nuevoNombre `
                 -OUPath $ouPath `
                 -Credential $credenciales `
                 -Restart `
                 -ErrorAction Stop
    Write-Host "¡Éxito! El equipo se reiniciará en unos segundos." -ForegroundColor Green
}
catch {
    Write-Host "Error al procesar la solicitud: $_" -ForegroundColor Red
}

🔍 ¿Qué hace este script exactamente?

  • Get-Credential: Evita que escribas la contraseña en el script. Abre la ventana típica de Windows para loguearte de forma segura.

  • -NewName: Cambia el nombre local del equipo al mismo tiempo que lo registra en el Active Directory. Esto evita el conflicto de nombres que tuviste hoy.

  • -ErrorAction Stop: Si algo sale mal (ej. contraseña incorrecta o el DC no responde), el script se detiene y te muestra el error exacto en rojo en lugar de intentar forzar el reinicio.

  • -Restart: Aplica los cambios de inmediato. Sin reinicio, el cambio de nombre no es efectivo y la relación de confianza no se termina de establecer.

💡 Un consejo de "pro"

Si alguna vez tienes que hacer esto en muchos equipos a la vez, puedes guardar una lista de nombres en un archivo .csv y hacer que el script los recorra uno a uno.

Pero...

Pero... ¿Cómo puedo validar que esto quedó bien y funcionando

¡Buena pregunta! En soporte, no asumimos que funciona hasta que lo probamos. Una vez que el equipo se reinicia tras ejecutar el script, existen tres niveles de validación para dormir tranquilo.

Validación

Aquí tienes el "checklist" de verificación:

1. Validación Local (La prueba de fuego)

La forma más rápida de saber si la relación de confianza es sólida es intentar loguearse.

  • Inicio de sesión: Intenta entrar con una cuenta de dominio que nunca haya iniciado sesión en esa máquina. Si el perfil se crea y entras al escritorio, el canal seguro está perfecto.

  • Consola (PowerShell): Ejecuta este comando para que Windows mismo testee la conexión con el DC:

    •                             Test-ComputerSecureChannel

    • Si devuelve True, todo está en orden.

    • Si devuelve False, algo falló en la autenticación.

    2. Validación en el Active Directory (AD)

    Desde el servidor o un equipo con las herramientas de administración (RSAT) instaladas:

    • Usuarios y Equipos de AD: Busca el nombre del equipo. Verifica que:

      1. El objeto existe en la OU correcta (si usaste el parámetro -OUPath).

      2. En la pestaña "Attribute Editor" (o en General), la propiedad dNSHostName coincida con el nuevo nombre que asignaste.

    • DNS Manager: Asegúrate de que se haya creado un registro tipo A con el nuevo nombre del equipo apuntando a su IP actual.

    3. Validación de Políticas (GPO)

    Un equipo unido al dominio debe recibir las directivas de la empresa. Para verificar que las está "escuchando", usa:

    gpresult /r

    En el apartado "Configuración del equipo", deberías ver el nombre de tu dominio y las GPOs aplicadas. Si ves un error que dice "El usuario no tiene datos de RSoP", es probable que necesites forzar una actualización con gpupdate /force.

    📊 Resumen de Comandos Rápidos

    Qué quieres saber Comando (PowerShell/CMD) Resultado esperado
    ¿Está el canal activo? Test-ComputerSecureChannel True
    ¿Quién soy en la red? whoami /fqdn El nombre completo en el dominio
    ¿A qué dominio pertenezco? (Get-WmiObject Win32_ComputerSystem).Domain dominio.com
    ¿Tengo mis políticas? gpresult /r Lista de GPOs aplicadas
    Tip

    Si el comando Test-ComputerSecureChannel te devuelve: True pero el usuario sigue viendo errores de confianza, a veces el reloj del equipo está desincronizado con el del servidor (más de 5 minutos de diferencia). Asegúrate de que la hora sea idéntica.

    🏁 Conclusiones y Mejores Prácticas: Gestión de Dominio

    La administración de equipos en un dominio mediante CLI no es solo una cuestión de velocidad, sino de consistencia y reducción de errores humanos. Tras analizar los métodos, el error de confianza y la validación, estas son las claves maestras:

    1. La Herramienta Ideal

    • PowerShell es el estándar actual: El comando Add-Computer es superior a netdom porque permite gestionar el nombre del equipo (-NewName) y su ubicación organizativa (-OUPath) en una sola línea de ejecución.

    • Automatización Segura: Siempre utiliza Get-Credential o el asterisco (*) para el manejo de contraseñas. Nunca dejes credenciales en texto plano dentro de scripts.

    2. El Error de Confianza (Trust Relationship)

    • Diagnóstico: El error suele ser un síntoma de un "desfase" entre el secreto guardado en el equipo local y el objeto en el Active Directory.

    • Causa Raíz: En tu caso, la duplicidad de nombres es el culpable número uno. Al usar el mismo nombre, el AD "pisa" la cuenta anterior, invalidando el acceso del equipo original.

    • Solución Ágil: Antes de desunir y volver a unir (el camino largo), intenta siempre Test-ComputerSecureChannel -Repair. Es el "primer auxilio" del administrador de redes.

    3. El Ciclo de Vida del Soporte (Checklist)

    Para asegurar que un equipo quedó correctamente integrado, sigue siempre este flujo:

    1. Conectividad: Validar DNS y visibilidad del DC.

    2. Ejecución: Script de unión con cambio de nombre único.

    3. Verificación: * Test-ComputerSecureChannel para validar el canal.

      • gpresult /r para confirmar la recepción de políticas.

      • Revisión de la hora del sistema (debe coincidir con el DC).

    Tip

    💡 Nota Final del Especialista

    El éxito en el soporte técnico de infraestructura no está en saberse todos los comandos de memoria, sino en limpiar el entorno antes de actuar. Si mantienes tu Active Directory libre de objetos antiguos y nombres duplicados, el 90% de estos errores desaparecerán.
